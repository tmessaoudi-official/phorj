# Phorge M2.5 — Phase 2 (cross-OS standalone builds) — Design

> Implementation-level refinement of **Phase 2** from the frozen master design
> (`docs/specs/2026-06-16-m2.5-phorge-build-design.md`, §4/§5/§7 + decisions B-1…B-11). The
> macro-architecture — payload = named **section**, versioned CRC-guarded **container**, hand-rolled
> std-only reader, stub model — is settled there and is **not** re-litigated here. This doc fixes the
> module layout, the per-format reader algorithms, the cross-build orchestration, the cache, the CLI
> surface, and the test strategy for the Phase-2 slice. Phase 1 (`x86_64-linux-gnu`) is complete and
> tagged `v0.3.0`.

## 1. Goal & scope of this slice

**Goal.** `phorge build foo.phg --target <triple>` (and `--all`) produces a standalone native
executable for **Linux (x86_64-musl, aarch64-gnu, aarch64-musl)** and **Windows (x86_64-pc-windows-gnu)**
by cross-compiling a runtime *stub* locally (zig as the linker), caching it, and embedding the program
source as a named section — the same mechanism as Phase 1, generalized across object formats.

**In scope.**
- `bundle/` module split; the format-dispatching `find_section` + **PE**, **Mach-O**, **fat/universal**
  reader arms (all std-only, hand-rolled, checked arithmetic).
- On-demand `cargo-zigbuild` stub build + a local **stub cache** keyed on an FNV-1a-64 of the host
  phorge binary.
- `--target <triple>` / `--all` CLI, output-path scheme, `--cap-lints=warn` for the stub build, and a
  precise missing-`rustup`-target error.

**Out of scope (deferred — §8).** macOS stub cross-builds + ad-hoc signing (the Mach-O/fat *reader*
ships and is fixture-tested now; producing the Mac *stub* waits on `rcodesign` + a macOS SDK); all of
Phase 3 (CI stub registry, download-and-cache, Authenticode/notarization, `--sign`); qemu/wine
execution of aarch64/windows artifacts.

## 2. Module restructure — `bundle.rs` → `bundle/`

Phase 2 roughly triples the module (three more readers + dispatch + orchestration + cache). A single
700+-line file mixing three binary formats, CRC logic, and subprocess orchestration does too much.
Promote to a directory module of focused, independently-testable units:

| File | Responsibility |
|---|---|
| `bundle/mod.rs` | Public surface: `embedded_source()`, `SECTION_NAME`/`MACHO_SEG`/`MACHO_SECT`, re-exports. |
| `bundle/container.rs` | `MAGIC`, `CONTAINER_VERSION`, `HEADER_LEN`, `crc32`, `encode_container`, `decode_container` — **moved verbatim** from today's `bundle.rs` (behavior-preserving; existing container tests move with it). |
| `bundle/section.rs` | `find_section(bytes) -> Option<&[u8]>` — the magic-sniffing dispatcher; finds the phorge payload under each format's canonical name(s). |
| `bundle/elf.rs` | `elf_find_section` — **moved verbatim** from today's `bundle.rs` (incl. its adversarial test). |
| `bundle/pe.rs` | `pe_find_section` + synthetic-fixture tests. |
| `bundle/macho.rs` | `macho_find_section` (thin Mach-O) + `fat_find_section` (universal wrapper) + tests. |
| `bundle/cross.rs` | Cross-build orchestration + cache (the `cmd_build` body moves here from `cli.rs`; `cli::cmd_build` becomes a thin shim). Named `cross.rs`, not `build.rs`, to avoid confusion with Cargo build scripts and `tests/build.rs`. |

Per-format helpers keep their natural signatures (`elf_find_section(bytes, name)`,
`pe_find_section(bytes, name)`, `macho_find_section(bytes, seg, sect)`, `fat_find_section(bytes, seg,
sect)`) for testability; the public `find_section(bytes)` sniffs the format and calls the right helper
with the phorge constants (`.phorge` for ELF/PE, `__PHORGE,__source` for Mach-O). The module split is
the only structural change; logic that moves is unchanged.

## 3. Format dispatch + the reader arms

`find_section` sniffs the leading magic and delegates; **unknown magic → `None` → caller falls through
to the normal CLI** (never panics). All readers do **minimal section lookup, not full parsing** (master
spec §6/§9) and use `checked_add`/`checked_mul` on every offset (EV-7 — adversarial input returns
`None`, never overflow-panics under the debug/test profile). Field offsets below are the implementation
reference; each is pinned against a fixture during TDD.

```
magic @0          format        delegate
7F 45 4C 46       ELF           elf_find_section   (exists)
4D 5A  ("MZ")     PE/COFF       pe_find_section
CF FA ED FE       Mach-O 64 LE  macho_find_section (MH_MAGIC_64 = 0xFEEDFACF, on-disk LE)
CA FE BA BE       fat/universal fat_find_section   (FAT_MAGIC, big-endian header)
else              —             None
```

**PE/COFF** (`bundle/pe.rs`) — [Inferred: PE/COFF spec; pinned by fixture in TDD]
- DOS header: `MZ` @0; `e_lfanew` (u32 LE) @ `0x3C` → offset of the `PE\0\0` signature.
- COFF File Header (20 B) immediately after the 4-byte signature: `NumberOfSections` (u16) @ +2,
  `SizeOfOptionalHeader` (u16) @ +16.
- Section table starts at `e_lfanew + 4 + 20 + SizeOfOptionalHeader`. Each section header is 40 B:
  `Name[8]` @0 (null-padded; `.phorge` is 7 chars, so the simple 8-byte-name path suffices — no
  `/<decimal>` string-table indirection needed), `SizeOfRawData` (u32) @16, `PointerToRawData` (u32) @20.
- Match `Name == ".phorge"` → return `bytes[PointerToRawData .. +SizeOfRawData]`.

**Mach-O 64** (`bundle/macho.rs`) — [Inferred: Mach-O ABI; pinned by fixture in TDD]
- `mach_header_64` (32 B): `magic` @0, `ncmds` (u32) @16. Load commands start @32.
- Each load command: `cmd` (u32) @0, `cmdsize` (u32) @4; iterate `ncmds`, advancing by `cmdsize`.
- `LC_SEGMENT_64` = `0x19`: `segment_command_64` header is 72 B (`segname[16]` @8, `nsects` (u32) @64),
  followed by `nsects` × `section_64` (80 B each): `sectname[16]` @0, `segname[16]` @16, `size` (u64)
  @40, `offset` (u32) @48.
- Match `segname == "__PHORGE"` && `sectname == "__source"` → return `bytes[offset .. +size]`.

**Fat/universal** (`bundle/macho.rs`) — [Inferred: fat ABI; pinned by fixture in TDD]
- `fat_header` (big-endian): `magic` @0, `nfat_arch` (u32) @4. `fat_arch` (20 B, BE): `cputype` @0,
  `offset` (u32) @8, `size` (u32) @12.
- Iterate slices; recurse `macho_find_section` into each slice (`bytes[offset .. +size]`) until the
  section is found. (Self-read on macOS: `current_exe()` returns the *fat* file; pick the slice matching
  the running `cputype`. Phase 2 produces at most thin Mach-O, deferred — the fat arm is built +
  fixture-tested, forward-looking.)

**`embedded_source()` dispatches via `find_section`** (sniffs the running image's own format), so a
produced Windows `.exe` self-reads its PE section and a Mac binary self-reads Mach-O — which is why all
arms ship now even though the macOS *stub* is deferred.

## 4. Cross-build orchestration (`bundle/cross.rs`)

Phase 2 has **no CI** (that's Phase 3), so the model is **build the stub locally on demand, then cache**
— the natural precursor to Phase 3's download-and-cache (same cache, same embedding step).

`phorge build <file> [-o <out>] [--target <triple> | --all]`:
1. **Validate** the program (`cmd_check` — lex→parse→check); on error, diagnostics + exit 1, emit
   nothing (unchanged from Phase 1).
2. **Resolve targets** — host default · `--target <triple>` · `--all` = host + the four:
   `x86_64-unknown-linux-musl`, `aarch64-unknown-linux-gnu`, `aarch64-unknown-linux-musl`,
   `x86_64-pc-windows-gnu`.
3. **Per target T:**
   - **Cache lookup** (§5). On miss: ensure the rustup target is installed (else precise error, below),
     then `cargo-zigbuild build --release --target T` with `RUSTFLAGS=--cap-lints=warn` **on the
     subprocess only** (so the tracked `[lints] warnings="deny"` can't fail a cross build on a
     target-specific warning — no manifest edit). The built `target/T/release/phorge[.exe]` is the
     **stub** (a phorge with no embedded section → `embedded_source()` → `None` → normal CLI). Copy it
     into the cache.
   - **Embed**: copy cached stub → output path; `llvm-objcopy --add-section <name>=<container>
     --set-section-flags <name>=noload,readonly` (`name` = `.phorge` for ELF/PE, `__PHORGE,__source`
     for Mach-O); `chmod +x` on unix non-windows outputs. (`PHORGE_OBJCOPY` override honored, as Phase 1.)
4. **Output** — host / single `--target` → `-o` or `./<stem>` (`.exe` for windows); `--all` →
   `dist/<stem>-<target>[.exe]` per target.

**Driver** — `cargo-zigbuild` (host build-dep; `cargo install --locked cargo-zigbuild`), with `zig`
0.16.0 on `PATH`. It owns the rustc→zig linker plumbing, glibc floor pinning for gnu targets, and the
windows-gnu/musl link config — preferred over a bespoke `zig cc` wrapper (master spec §9 risk). The
invocation pins `--bin phorge` so the crate's single intended binary is built regardless of any future
additional `[[bin]]` targets.

**Source checkout required for cross targets (Phase 2 limit).** `cargo-zigbuild` compiles the phorge
crate *from source*, so `phorge build --target/--all` must run from a phorge **source checkout** (a
`Cargo.toml` present). A *distributed* phorge binary has no source tree and therefore cannot
self-cross-build in Phase 2 — `build_stub` detects the absent `Cargo.toml` and errors with a precise
message that points at Phase 3's download-a-prebuilt-stub model. (The **host** build needs no source: it
reuses the running binary itself as the stub, so a distributed phorge can still `build` for its own OS.)

**Missing target** — if `rustc --print target-list`/`rustup target list --installed` lacks T,
`phorge build` errors with `target '<T>' not installed — run: rustup target add <T>` (a precise,
actionable message, not a cryptic cargo failure; master spec §9 risk).

**`--sign`** — reserved; if passed in Phase 2, error `signing is Phase 3` (it's the opt-in
final-artifact step that needs `rcodesign`/`osslsigncode`).

## 5. The stub cache

Layout: `${XDG_CACHE_HOME:-$HOME/.cache}/phorge/stubs/<key>/<triple>/phorge[.exe]`.

`<key>` = **FNV-1a-64 (hex) of the running phorge binary's own bytes** (`std::env::current_exe()` →
`fs::read` → hash). Rationale (decision B-6 / [[m2-p3.5-complete-next-p4]]): the key must bind the
phorge identity so a rebuilt phorge invalidates stale cross-stubs — otherwise a stale stub embeds your
source into an **old VM**, silently breaking the parity spine (`docs/INVARIANTS.md` #1) at the
distribution layer. Hashing the binary's bytes means **any** source/VM change → different bytes →
different key → cache miss → rebuild.

FNV-1a-64 (offset basis `0xCBF29CE484222325`, prime `0x100000001B3`) is ~10 lines of std; a cache key is
not a security boundary, so collision-resistance beyond ~1/2^64 is unwarranted (a collision's blast
radius is local + recoverable: delete the cache). A hand-rolled SHA-256 was considered and rejected as
overkill for a non-security cache key (decision P2-3).

## 6. CLI surface (delta over Phase 1)

```
phorge build <file> [-o <out>] [--target <triple>] [--all]
```
- No `--target` / `--all` → host target (Phase-1 behavior preserved): `-o` or `./<stem>`.
- `--target <triple>` → single cross target → `-o` or `./<stem>[.exe]`.
- `--all` → host + the four targets → `dist/<stem>-<target>[.exe]`.
- `--target` and `--all` are mutually exclusive (error if both); unknown flags / extra args → usage,
  exit 2 (Phase-1 strict-arg behavior generalized).
- An apple/darwin `--target` is **rejected** (exit 1, "macOS stub deferred — §8") rather than silently
  producing a Mach-O with a mismatched `.phorge` section; macOS leaves the reader shipped but the stub
  deferred (F7 / §8).

## 7. Test strategy (3 tiers)

1. **Reader unit tests — offline, always run** (`#[cfg(test)]` in `bundle/pe.rs`, `bundle/macho.rs`).
   Synthetic hand-crafted fixtures: a minimal valid header with an injected section → `find_section`
   returns the right bytes → `decode_container` round-trips. Adversarial fixtures (overflow offsets,
   truncation, **wrong-endianness** — the new trap) → `None`, no panic (EV-7). Mirrors the existing ELF
   tests. This is the bulk of reader correctness; no toolchain required.
2. **Real-binary round-trip — toolchain-gated, graceful skip** (`tests/build.rs`). Per buildable target:
   `cargo-zigbuild` the stub → embed → `llvm-objcopy --dump-section` it back → assert
   `decode_container == source`. Skips with an `eprintln` if `cargo-zigbuild` / the rustup target is
   absent (same pattern as today's `elf_reader_finds_added_section`).
3. **Native execution — host-runnable targets** (`tests/build.rs`). `x86_64-unknown-linux-musl` produces
   an x86_64-Linux binary that runs on this box → full byte-identity vs `runvm` (Phase-1-style, across
   the gnu→musl libc boundary — a bonus cross-libc parity check). aarch64/windows can't execute here →
   covered by tier 2; if `qemu-user-static` is later present, tier 3 extends to aarch64 (detected at
   test time, not built now).

Gate unchanged: `cargo test` green, `cargo clippy --all-targets` + `cargo fmt --check` clean,
`#![forbid(unsafe_code)]` intact, **zero new runtime crate deps** (cargo-zigbuild/zig/llvm-objcopy are
host build-tooling, outside the std-only line per master spec §6).

## 8. Deferred / out of scope

- **macOS stubs + signing** — Mach-O/fat *reader* ships + fixture-tested; producing the Mac *stub* waits
  on `rcodesign` (ad-hoc `-s -`) + a macOS SDK for zig to link against. When that lands, the Mach-O/fat
  offsets must get a real-binary tier-2 round-trip (the fixture-leakage caveat in §10).
- **Phase 3** — CI stub registry, download-and-cache, Authenticode (EV cert), Developer-ID + notarize +
  staple via `rcodesign` from Linux, `--sign`.
- **qemu/wine execution** of aarch64/windows artifacts — not assumed; tier 2 (dump round-trip) covers
  them; tier 3 extends opportunistically if qemu appears.

## 9. Decisions Log

| # | Decision | Choice | Rationale |
|---|---|---|---|
| P2-1 | Phase-2 scope | **Full pass**, but **Linux + Windows real stubs now, macOS reader-ready/stub-deferred** | macOS leg needs `rcodesign` + a macOS SDK (materially heavier); the reader arms are free and load-bearing for the eventual Mac binaries |
| P2-2 | `bundle.rs` structure | Split → `bundle/` per-format modules + dispatcher | Three independent formats behind one `find_section` interface; each independently fixture-testable; avoids a 700+-line do-everything file |
| P2-3 | Stub-cache key | **FNV-1a-64 of the host phorge bytes** | Binds phorge identity so a rebuilt phorge invalidates stale stubs (B-6); a cache key isn't a security boundary, so SHA-256 is overkill |
| P2-4 | Stub provisioning (Phase 2) | **Build locally on demand via `cargo-zigbuild`, then cache** | No CI yet (Phase 3); natural precursor — same cache + embedding as the download model |
| P2-5 | Cross-build driver | `cargo-zigbuild` + zig 0.16.0 | Owns rustc→zig linker plumbing, glibc pinning, windows-gnu/musl; preferred over a bespoke wrapper (master §9) |
| P2-6 | Deny-lints vs cross builds | `RUSTFLAGS=--cap-lints=warn` on the stub-build subprocess only | Tracked `[lints] warnings="deny"` can't be relaxed per-target without editing `Cargo.toml`; cap on the subprocess leaves the manifest untouched |
| P2-7 | Output paths | host/single → `-o` or `./<stem>[.exe]`; `--all` → `dist/<stem>-<target>[.exe]` | Matches master spec §5; predictable, collision-free across targets |
| P2-8 | Reader robustness | Minimal section lookup + checked arithmetic on every offset; unknown magic → `None` | EV-7 (never panic on adversarial input); endianness flips (LE Mach-O vs BE fat) are the new trap surface |
| P2-9 | Cross stub provenance (Phase 2) | Build from a **source checkout** via `cargo-zigbuild --bin phorge`; a distributed (sourceless) phorge errors clearly on `--target/--all` | Phase 2 has no prebuilt-stub registry (that's Phase 3); the host build still works sourceless (reuses the running binary), so only *cross* targets carry the source-checkout requirement |
| P2-10 | `--all` host artifact naming | Name the host artifact `dist/<stem>-<host-triple>[.exe]` (resolved from `rustc -vV`), not `dist/<stem>-host` | One consistent `<stem>-<triple>` scheme across every `dist/` artifact; the literal `-host` was ambiguous and broke the otherwise triple-keyed naming |

## 10. Risks

- **Hand-rolled PE/Mach-O/fat readers** — mitigated by needing only *section lookup* (not full parsing),
  fully fixture-tested offline, and the real-binary dump round-trip for the targets buildable here (ELF
  host + PE via the C5 windows round-trip). **Mach-O/fat caveat:** with no Mac stub produced in Phase 2,
  the Mach-O/fat readers have **only** synthetic-fixture validation — a self-consistent-but-wrong offset
  shared by the fixture builder and the reader (fixture leakage) would pass unit tests yet fail on a real
  Mach-O binary. The macOS-stub session (deferred, §8) **must** re-verify the Mach-O/fat offsets against a
  real `cargo-zigbuild` Mach-O via a tier-2 dump round-trip before shipping Mac binaries.
- **Endianness** — fat headers are big-endian while Mach-O bodies are little-endian; the `u*at` helpers
  must be endianness-explicit per format. Adversarial wrong-endianness fixtures guard this.
- **`llvm-objcopy --set-section-flags` on PE/COFF (resolved — F5).** The `noload,readonly` flags are
  **required on PE**, not cosmetic: without them, `--add-section` writes a PE section header with **zero
  raw data** (`VirtualSize 0x0`, 0 bytes dumped) and the embedded program is lost. An earlier attempt to
  *skip* the flags on PE (on the unverified theory that COFF support was unreliable) was itself the bug.
  The flags work correctly on PE and remain the proven Phase-1 behavior on ELF, so `embed_section`
  applies them **unconditionally** for both. Verified end-to-end by the tier-2 windows round-trip
  (`tests/build.rs::cross_windows_section_round_trips`) against a real `cargo-zigbuild` PE — the
  fixture tests could not catch this (they don't invoke `llvm-objcopy`).
- **zig 0.16.0 is pre-1.0** — pinned; `cargo-zigbuild` insulates against `zig cc` triple/flag churn.
- **Network for `cargo install cargo-zigbuild` + `rustup target add`** — one-time provisioning; the
  missing-target path emits a precise actionable error.
- **`--cap-lints=warn` vs 1.96** — verify the env path actually suppresses the deny gate for a cross
  build before relying on it (a tier-2 test exercises a real cross build).
