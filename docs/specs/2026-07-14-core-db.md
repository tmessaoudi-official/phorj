# Core.Db — the enhanced-PDO database primitive (DEC-208) — FULL SPEC (LOCKED 2026-07-14)

Locked with the developer over ~10 AskUserQuestion design rounds (2026-07-13/14). This is the SSOT
for the `Core.Db` surface; per-round rulings + alternatives are in
`docs/research/full-audit/raw/C-decisions.md` (DEC-208, DEC-220-S3, DEC-221). Governing philosophy:
`Core.Db` is a **primitive**, not an ORM — richer + faster + safer + more correct than PHP's PDO, but
ORMs/builders/migrations stay **userland** (DEC-208, META-6).

## Status
- **SHIPPED (this session):** the dynamic path + typed-generic base + throwing constructors.
  `new Db("sqlite:…")` (throwing ctor, DEC-221) · `prepare`/`bind`(`?`)/`bindNamed`(`:name`) ·
  `query()`→`List<Row>` + `r.getInt/getString/getFloat/getBool` · `exec()`→affected-int ·
  `queryInto<T>()`→`List<T>` + `queryOneInto<T>()`→`T?` (contextual T, flat, strict by-name) ·
  catchable `DbError`. Backed by bundled `rusqlite` (SQLite), `db` feature, spine-quarantined.
- **TO BUILD:** the slices below — a large multi-slice, multi-session build.

## Build slices (each a gated-green, byte-identity-safe unit; dep-free ones first, drivers last)
- **B — typed-query completion**: nested hydration (by T shape, dotted `"order.total"` aliases, eager) +
  `queryScalar<T>` + `queryMap<K,V>` + column naming strategy. Extends the shipped `desugar_db`. Dep-free.
  **STATUS (2026-07-14): 4-of-4 SHIPPED** — nested hydration (arbitrary depth, optional-entity→null on
  all-NULL LEFT-JOIN via new `Row.isNull`; `E-DB-HYDRATE-CYCLE` rejects self-referential graphs),
  `queryScalar<T>` (exactly-one-row-and-column, else `DbError`; sole column read via new `Row.columnNames`),
  and `queryMap<K,V>` (key = first column `int`/`string`; value = second column scalar, or an entity
  hydrated by field name). Example `examples/db/nested.phg`; fixtures in `tests/db.rs`; both backends

  snake_case-DB ↔ camelCase-phorj mapping via a **compile-time, per-query method**:
  `stmt.namingStrategy(new Naming.SnakeToCamel()).queryInto()`. The `desugar_db` pass reads the strategy
  from the call chain AT COMPILE TIME and generates `getString("user_name")` for a `userName` field
  (the `snake_case` of the field name), applied per dotted segment for a nested alias
  (`"home_address.street_name"`). Zero runtime cost (`Statement.namingStrategy` is a chainable no-op
  returning `this`, present only so the chain type-checks); a runtime/non-literal argument is rejected
  (`E-DB-NAMING-NOT-CONST` — never a silent downgrade). `Naming` is a member-gated Core.Db enum (`Exact`
  default + `SnakeToCamel`; `import Core.Db.Naming;` — nothing in the wind). The strict-exact by-name
  default is unchanged. Applies to by-field-name hydration only (`queryInto`/`queryOneInto`, a
  `queryMap` entity value); `queryScalar` and scalar map values read by position and ignore it.
  **Rejected alternatives** (recorded per ADJUDICATION): (b) **runtime, connection-level**
  `db.namingStrategy(...)` — needs the strategy threaded onto the row/accessor at runtime (row is a
  plain `Map`); more machinery, no compile-time guarantee; (c) **type-directed** `#[Column("user_name")]`
  per field — most verbose. **Surface-shorthand note:** the ruling was written `Naming.SnakeToCamel`;
  realized as the type-honest `new Naming.SnakeToCamel()` (zero-payload variant construction requires
  `new`, matching `RoundingMode`) — developer to confirm. Example `examples/db/naming.phg`; fixtures in
  `tests/db.rs`; both backends (`run ≡ runvm`). The former failing program
  (`… .prepare("SELECT user_name FROM u").queryInto()` throwing `getString: no column \`userName\``) now
  succeeds under the strategy.
- **C — transactions/correctness**: `db.transaction(fn)` (auto-rollback) + savepoints (nested) +
  isolation levels + `using`/`Closable` (DEC-203) + typed `DbError` taxonomy (UniqueViolation/
  ConnectionError/Timeout/Deadlock/SerializationFailure/SyntaxError) + retry (needs the taxonomy). Dep-free.
  - **SHIPPED 2026-07-14** — manual PDO-faithful `begin`/`commit`/`rollback` (savepoint-aware) +
    `rollbackQuiet` (the `finally` auto-rollback idiom) + the typed `DbError` taxonomy + deterministic
    `close()`; AND (unblocked by DEC-222 throwing closures) the **closure form** `db.transaction(fn)`
    (COMMIT on normal return returning the closure's value; auto-ROLLBACK + re-throw the ORIGINAL typed
    error on a throw; a nested call is a SAVEPOINT) via a `HigherOrder` native that preserves the thrown
    value through the rollback (`pending_throw`), plus `db.transactionRetry(fn, retries)` (retries the
    transient `SerializationFailure` only; loop in the prelude so it can `catch` the TYPED error). BOTH
    the closure form and the manual control are supported (developer ruled BOTH). Example
    `examples/db/transaction-closure.phg`; fixtures in `tests/db.rs`; both backends (`run ≡ runvm`).
    **SURFACE — PENDING adjudication:** the spec (§5) illustrates one method `db.transaction(retries: N,
    fn)`, but the language has NO named args, NO method default params, and NO generic-method overloading
    — so a single generic `transaction` cannot carry an optional `retries`; realized as the distinct
    `db.transactionRetry(fn, retries)` (developer to confirm the name/shape). **DEFERRED:** `using`/
    `Closable` auto-close (DEC-203 language slice; `close()` shipped) and isolation levels (SQLite has
    ~one; meaningful once Postgres lands — isolation-arg retry rides with it). See DEC-208 slice-C in the
    decision register + `KNOWN_ISSUES.md`.
- **D — writes/robustness**: `lastInsertId()`/`RETURNING` + bulk `executeMany` + typed `IN`-list binding
  + query `.timeout(ms)` + `Db.onQuery` observability hook. Dep-free.
  - **SHIPPED 2026-07-14** — `db.lastInsertId(): int` + `stmt.execReturningId(): int` (SQLite
    `last_insert_rowid()`); `stmt.executeMany(rows): int` (prepare-once, atomic under one `phorj_bulk`
    savepoint, total affected); `stmt.bindList(values): Statement` (the single `?` in `… IN (?)` expands
    in place to a comma list — empty → `IN (NULL)`; interleaves left-to-right with `bind()`;
    `?`-in-string-literal skipped); `db.timeout(ms): Db` (SQLite `busy_timeout` — bounds LOCK-WAIT only,
    not a CPU-bound runaway; scope documented); `db.onQuery((string, int) => void): Db` (hook invoked
    after each query/exec — `query`/`exec`/`executeMany`/`execReturningId` are `HigherOrder` natives that
    call back into the backend to run the stored closure). Example `examples/db/writes.phg`; fixtures in
    `tests/db.rs`; both backends (`run ≡ runvm`; the hook's wall-clock `ms` is excluded from any
    byte-identity output). **Notes / deviations:** (1) `bindList`/`executeMany` are GENERIC over the
    element type (`bindList<T>`/`executeMany<T>`), NOT `List<bindable-union>` — an invariant `List<union>`
    cannot accept a homogeneous list argument, so bindability is enforced at RUNTIME (`to_sql` → catchable
    `DbError`) not compile-time. (2) `executeMany` rows are homogeneously-typed lists (phorj list literals
    must share one element type); a mixed-column row needs a per-row typed binding (`List<string|int> r =
    [1,"x"];`). (3) With `db.timeout(ms)` armed, a transient `busy`/`locked` is reclassified
    `SerializationFailure` → `Timeout` — so `SerializationFailure` (the class a future closure-`retry`
    targets) is not observed while a timeout is set; acceptable while retry stays deferred.
- **E — value mapping**: DB↔`enum` + numeric→`decimal` + timestamp→`DateTime` (gated on DEC-206) +
  JSON→`Core.Json`. Dep-free.
  - **SHIPPED 2026-07-14 (enum + decimal + JSON; DateTime DEFERRED)** — three column→type conversions
    the `queryInto`/`queryOneInto` desugar performs at COMPILE TIME, composing with the flat/nested/
    optional shapes (a nested entity's enum/decimal/Json field hydrates too):
    (1) **enum** — a hydrated field whose type is a phorj `enum` maps from a **TEXT** column by matching
    the column value against the variant **NAME**, case-sensitively (`'Active'` → `Status.Active()`).
    **ZERO-payload variants only** — an enum with any data-carrying variant is a compile error
    (`E-DB-HYDRATE-ENUM-PAYLOAD`); an unknown column value is a catchable `DbError` (strict, no silent
    coercion); an `enum?` field admits NULL. Lowered to `match (row.getString("col")?) { "V" =>
    new Enum.V(), … default => DbError.fail(…)? }` (an optional field wraps it in a `mutable … = null;
    if (!row.isNull("col")?) { … }`, the same idiom as an optional nested entity).
    (2) **decimal** — a `decimal`/`decimal?` field maps via a new `Row.getDecimal`/`getDecimalOrNull`
    accessor. **Exact money, never float**: a TEXT column is parsed with the exact `…d`-literal grammar
    (`value::decimal_of`), so `'0.10'` is exactly `0.10`; an INTEGER is exact at scale 0; a REAL is
    converted through its shortest round-trip decimal string (best-effort — **store decimal columns as
    TEXT** for guaranteed exactness). NULL-into-non-optional / a non-decimal value → `DbError`.
    (3) **JSON** — a `Json`/`Json?` field (Core.Json's injected enum) maps from a TEXT column by parsing
    via `Json.parse` (the program's own `import Core.Json` makes `Json` nameable — nothing in the wind);
    invalid JSON → `DbError`; `Json?` admits NULL. Lowered to `Json.parse(row.getString("col")?) ??
    DbError.fail(…)?`.
    **DEFERRED — timestamp → `DateTime`**: gated on DEC-206, which has not built `DateTime`; not
    implemented (see `KNOWN_ISSUES.md`). **Boundary**: enum/Json are field-level mappings, not scalar
    column accessors, so a *direct* `queryScalar<Status>()` / `queryMap<_, Status>()` (a bare enum/Json
    as the whole scalar/value) is out of scope (a scalar column has no way to name its target enum) —
    `decimal` IS a scalar and works as a direct `queryScalar<decimal>` / map value. Example
    `examples/db/mapping.phg`; fixtures in `tests/db.rs`; both backends (`run ≡ runvm`).
- **F — compile-time safety**: SQL-injection lint (`W-SQL-INJECTION`) + placeholder/bind arity check. Dep-free.
  - **PARTIALLY SHIPPED 2026-07-14 (SQL-injection lint; arity check DEFERRED).** The `W-SQL-INJECTION`
    compile-time lint is shipped — a type-directed, `Core.Db`-import-gated warning on

    **The placeholder/bind ARITY check is DEFERRED** — a sound (no-false-positive) `?`-vs-`.bind()` check is
    not cheaply buildable: the zero-bind runtime path counts via rusqlite's comment/numbered-aware parser

    throws a catchable `DbError` on any mismatch, so it is a convenience lint, not a correctness gap — see

- **G — credential safety**: DSN password as `Core.Secret` (redacted everywhere). Small.
  - **SHIPPED 2026-07-14** — `Db.withPassword(string dsn, Secret<string> password): Db throws DbError`,
    a static factory (the sole `Db` constructor takes a `dsn`, and phorj has no constructor overloading,
    so the credential path is a factory over `new Db(...)`). The password is injected into the DSN by the
    pure `DbSys.dsnWithPassword` native (percent-encoded into the `postgres://user:PW@host` authority) ONLY
    at the connect boundary, then the driver parses it back OUT into its connection config and stores just
    a REDACTED DSN — so the plaintext is never retained on the handle and a connect error prints the host
    but never the password (unlike PDO, which leaks the DSN in exceptions). The redaction is structural
    (nothing to scrub after the fact) and additionally the connect-time diagnostic uses a
    `redact_dsn_password`-scrubbed DSN (URL and keyword forms). SQLite has no password → the DSN is passed

## 1. Connection & drivers
- **Multi-driver architecture** — `new Db(dsn)` dispatches on the DSN scheme behind a `DriverConn` trait
  (SHIPPED 2026-07-14, slice I): `sqlite:…` (rusqlite, shipped) · `postgres://…` (the sync `postgres`
  crate, `db-postgres`, SHIPPED) · `mysql://…` (MariaDB = MySQL protocol, slice J, not yet). Each driver
  is a vetted, feature-gated, **sync** crate (async/tokio stays disallowed at the phorj-facing API per the
  dependency policy — the `postgres` crate's internal tokio blocking-wrapper is its own impl detail),
  spine-quarantined + fixture-tested. Oracle deferred (no clean pure-Rust sync driver). Architecture is
  **expandable** — a new backend is one `DriverConn` impl + one dep admission (`src/native/db/<backend>.rs`).
- **Throwing constructor** (DEC-221): `new Db(dsn) throws DbError` (fail-fast, PHP-`new PDO`-faithful).
- **Credential safety**: the password is a `Core.Secret` so it is **redacted in every error / log /
  toString** — a connect error prints host but never the password (PDO leaks the DSN in exceptions).
- **`using`/`Closable`** (DEC-203): `using (Db db = new Db(dsn)) { … }` closes the connection
  deterministically (RAII) even on early return/throw.

## 2. Statements & binding
- **Prepared-first**: `db.prepare(sql)` → `Statement`; every path goes through a statement.
- **Both bind styles**: positional `.bind(v)` (`?`) and named `.bindNamed("n", v)` (`:name`), mutually
  exclusive per statement, chainable, typed (`string | int | float | bool | null | bytes`).
- **Typed `IN`-list binding**: `.bindList([1,2,3])` on `… IN (?)` auto-expands to `(?,?,?)` with each
  value bound — strictly better + safer than PDO (which can't bind arrays to `IN`).
- **SQL-injection compile-time lint**: a variable string-interpolated into a `prepare(...)` SQL literal
  (`prepare("… {userId}")`) is flagged (`W-SQL-INJECTION`) and steered to a bind — kills injection at
  compile time (PDO can't).
- **Compile-time placeholder/bind arity check** (literal SQL): `?` count vs `.bind()` count → compile
  error on mismatch (best-effort for dynamic chains).

## 3. Results — hydration
- **Dynamic path** (no class): `query()` → `List<Row>`, `r.getInt/getString/getFloat/getBool("col")`
  (+ nullable variants), typed + strict (no silent coercion). The no-class escape hatch.
- **Typed-generic path**: `queryInto<T>()` → `List<T>`; `queryOneInto<T>()` → `T?` (0→null, 1→obj,
  >1→`DbError`). T specified either **contextually** (`List<User> u = …queryInto();`, shipped) OR via
  **turbofish** (`…queryInto<User>();`) — turbofish is a general language feature to add (method type
  args). Mapping is **by field NAME, STRICT**: missing col / type mismatch / NULL-into-non-optional →
  `DbError`; `T?` fields admit NULL; extra columns ignored.
- **Flat OR nested — chosen by T's SHAPE** (one generic call, no flag): a **scalar** field → its column
  (flat); an **entity** field → recursively hydrated from prefix-aliased columns (nested), the alias
  convention being **dotted** `"order.total"` (quoted identifier); an **optional entity** field
  (`Order? order`) → null when all its columns are NULL (LEFT JOIN). Flat is the zero-config default.
  **EAGER only** — nested is filled from the JOIN in ONE query (zero N+1). Lazy relation loading is
  REJECTED in the primitive (N+1, live-connection-bound stateful objects, ORM-territory) — it's a
  userland ORM concern.
- **`queryScalar<T>()`** — one typed value (`SELECT COUNT(*)`/`MAX`). **`queryMap<K,V>()`** — index rows
  into a `Map<K,V>` by a key column (id→entity).
- **Lazy streaming**: `streamInto<T>()` yields `T` row-by-row (constant memory for huge results) — the
  real "laziness" (result streaming), distinct from lazy relations (which we don't do).
- **Column naming strategy**: configurable snake_case-DB ↔ camelCase-phorj mapping (`user_name`→
  `userName`) so you don't alias every column; strict-exact stays the default.
- **Value mapping**: DB value ↔ phorj `enum` (a `status` column → an enum, strict); numeric →
  `decimal` (exact **money**, never float); timestamp → `DateTime` (gated by DEC-206); JSON/`jsonb` →
  a phorj type via `Core.Json`; Postgres array (`int[]`) → `List<T>`.

## 4. Writes
- `exec()` → affected-row count. **`lastInsertId()` / `RETURNING`** — the auto-generated PK after an
  INSERT (essential for autoincrement/serial keys). **Bulk `executeMany(rows)`** — one prepared
  statement, many bind-sets (far faster than a loop).

## 5. Transactions
- `db.transaction(() => { … })` — commit on normal return, **auto-rollback on throw**. **Savepoints**:
  a nested `transaction` becomes a savepoint (composable partial rollback). **Isolation levels**:
  `db.transaction(Isolation.Serializable, fn)` (ReadCommitted/RepeatableRead/Serializable). **Retry**:
  `db.transaction(retries: N, …)` auto-retries transient `Deadlock`/`SerializationFailure` (Serializable
  is only practical with retry).

## 6. Errors — typed taxonomy
- Catchable `throws DbError` (never PDO's silent false/null). **Subtyped**: `UniqueViolation`,
  `ConstraintViolation`, `ConnectionError`, `Timeout`, `SerializationFailure`/`Deadlock`, `SyntaxError`
  (+ the underlying driver code). Callers `catch (UniqueViolation e)` vs `catch (ConnectionError e)` etc.
  Load-bearing: **retry** detects `Deadlock`/`SerializationFailure`; **timeout** produces `Timeout`.

## 7. Observability & robustness
- **`Db.onQuery((sql, ms) => …)`** — per-query hook for logging / metrics / slow-query / tracing.
- **Query timeout**: `.timeout(ms)` → a runaway query fails with `DbError.Timeout` instead of hanging.

## 8. Spine / LADDER
- LADDER case-1 (faithful → PHP PDO for transpile). Execution is `pure:false` → spine-quarantined from
  the byte-identity differential (live DB I/O ≠ byte-identical rusqlite-vs-PDO); `run ≡ runvm` holds
  (shared eval bodies); correctness via `tests/db.rs` fixtures on both Rust backends.

## Out of scope (userland or a separate wave — NOT the primitive)
- **Userland library (ORM/builder):** relations/lazy loading, unit-of-work, migrations, `upsert`/
  query-builder DSL, soft-deletes, optimistic locking.
- **Infra/userland:** connection pooling, read-replica routing, query-result caching.
- **Separate future wave:** sqlx-style **compile-time SQL-vs-schema checking** (needs a schema source).
- **Dialect-advanced, later:** Postgres `COPY`/pipeline/`LISTEN`/`NOTIFY`.
