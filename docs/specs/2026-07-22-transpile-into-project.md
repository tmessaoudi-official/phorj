# DEC-320 — `transpile-into-project`: file-by-file phorj adoption inside a live PHP app

**Status: v1 BUILT (2026-07-22) with the DEC-329 ruled defaults — see the DEC-320 build note in
the decision register for the shipped shape and the two disclosed deltas (classmap autoloader
supersedes the host-PSR-4 coupling; the F2 `phpInterop` namespace-prefix knob is deferred as a
PENDING adjudication — v1 keeps package path = namespace).**
Ruled scope (DEC-319/320, 2026-07-22): COMPILE-TIME ONLY — the TS→JS playbook. `.phg` files emit
`.php` siblings inside an existing PHP application so a team migrates one file at a time while the
app keeps running on PHP. The earlier live-interop rejection (on-the-spot rebuild, per-file gradual
typing) STANDS untouched.

## Why this is the #1 adoption lever (META-7 survey)

TypeScript did not win by being better than JavaScript in a vacuum — it won because `tsc` dropped
`.js` files exactly where the build already expected them (`allowJs`, in-place emit, later
`outDir`), so a million-line app could adopt it one file per pull-request. Kotlin did the same
inside javac projects; Swift inside ObjC targets (bridging headers). The common shape: **the new
language's output lands in the old build's addressing scheme, and the old code is importable from
the new code from day one.** For PHP the addressing scheme is composer's PSR-4 autoloader —
namespace path ≡ directory path.

## The five design forks

### F1 — Output placement (RECOMMENDED: sibling emit)
```text
src/Billing/Invoice.phg  ──phg──▶  src/Billing/Invoice.php   (same dir, same basename)
```
- **(a) SIBLING emit (recommended)**: the generated `.php` sits next to its `.phg`. The class's
  PSR-4 autoload path is UNCHANGED, so **zero composer.json edits** — the killer property, exactly
  the `tsc` in-place move. Generated files carry a first-line marker
  (`/* @generated by phg — edit the .phg */`) and a recommended `.gitattributes` `linguist-generated`
  entry; whether they are committed or ignored is the team's policy (both work — committed keeps
  deploys phorj-free, ignored keeps diffs clean + CI runs `phg build --php`).
- (b) OUT-DIR emit (`build/php/**`) + a composer PSR-4 mapping to the build dir: cleaner tree, but
  every adopting team must edit composer.json and reason about two roots — the exact friction (a)
  eliminates. Kept as a flag later if demanded; not v1.

### F2 — Package ↔ namespace ↔ directory mapping (RECOMMENDED: one config knob)
The app already owns a namespace root (`App\` → `src/`). phorj's `package Billing;` must land as
the app's `App\Billing`. Proposal: `phorj.json` grows an interop block —
```json
{ "phpInterop": { "namespaceRoot": "App", "sourceRoot": "src" } }
```
Rule: under `sourceRoot`, directory path = phorj package path = PHP namespace path prefixed by
`namespaceRoot` (the existing folder=package law extends naturally; E-PKG-PATH already enforces the
directory half). No per-file overrides in v1 — one law, loudly checked.

### F3 — Rebuild trigger (RECOMMENDED: explicit one-shot; watcher later)
- **(a) `phg build --php [paths…]` (recommended)**: one-shot, deterministic, CI-friendly —
  transpiles every project `.phg` (or the given subset) into its sibling `.php`, skipping files
  whose output is already current (mtime+hash). Composer install-time hooks stay REJECTED
  (supply-chain stance, recorded).
- (b) `phg watch --php`: dev-loop sugar over (a); queued as the v2 slice, not v1.

### F4 — Typing the not-yet-migrated PHP (RECOMMENDED: shipped `declare` interop now, stub GENERATOR next)
A `.phg` file importing a class that is still PHP needs a type surface. The mechanism ALREADY
exists: M8.5 foreign interop (`declare class Cart { function total(): float; }` /
`declare function legacy_tax(float x): float;`) — foreign refs emit `\Cart` global forms and are
never re-declared. v1 = write declare-blocks by hand (works today). v2 slice = **`phg stubs`**, the
`.d.ts` analog: generate declare-blocks from PHP sources' signatures (reflection-free, parser-based;
untyped params surface as adjudicated `mixed`-policy errors, never guesses).

### F5 — The shared runtime problem (MUST solve in v1: one runtime file)
Single-file transpile embeds the gated `__phorj_*` helpers per file. Two generated files loaded by
one request would fatally re-declare them. v1 REQUIREMENT: `phg build --php` additionally emits ONE
`src/_phorj/runtime.php` (every helper, function_exists-guarded or emitted once) and registers it
via composer `"files"` autoload — the only composer.json touch, done once by `phg init --php`
(printed as a diff for the user to apply; phg never edits composer.json itself — same
hand-the-command discipline as classifier-blocked commits).

## Contract with the spine

Per-file byte-identity holds exactly as today: a pure `.phg` compared under the differential; impure
modules keep their quarantines/content-parity tests. The generated sibling is REGULAR transpile
output — this feature adds placement + the shared runtime, never new semantics (Inv-5 untouched).

## Adjudications queued for the developer (each with the recommended default first)

1. **F1 placement**: sibling emit (recommended) vs out-dir?
2. **F2 knob shape**: `phpInterop { namespaceRoot, sourceRoot }` in phorj.json (recommended) — or a
   separate file?
3. **F5 runtime**: one `_phorj/runtime.php` + composer `files` entry (recommended) vs
   `function_exists` guards per file (no composer touch, but N copies of every helper)?
4. **Generated-file policy default in docs**: recommend committing them (deploys stay phorj-free)
   or ignoring them (clean diffs)? (Docs guidance only — both always work.)
5. **v1 cut line**: build `phg build --php` + F5 runtime now, `phg stubs` + `phg watch` as
   follow-ups (recommended) — or hold for the full set?
