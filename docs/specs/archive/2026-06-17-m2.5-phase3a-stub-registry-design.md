## 1. Goal & scope of this slice

**Goal.** A **distributed** phg binary (no source checkout) can `phg build foo.phg --target
<triple>` / `--all` for the Phase-2 cross targets by **downloading a prebuilt runtime stub** from a CI
registry, verifying it against a **baked sha256 manifest**, caching it, and embedding the program source
— the same `embed_section` step as Phases 1/2. This closes the Phase-2 limitation (decision P2-9): today
a sourceless phorj errors on `--target/--all` with *"wait for Phase 3's prebuilt stub download."*

**In scope.**
- The **download-and-cache** client: a third branch in `bundle/cross.rs::build_stub`'s cache-miss path.
- A hand-rolled std **SHA-256** (`bundle/sha256.rs`) + a **manifest** module (`bundle/manifest.rs`):
  parse, lookup, registry-base resolution, the baked-manifest seam.
- A **`build.rs`** that bakes the per-target sha256 manifest into the released primary binary.
- The **CI workflow** (`.github/workflows/release.yml`): build stubs → hash → bake → publish.
- `Cargo.toml`: add `repository` (single-sources the registry base).

**Out of scope (deferred to Phase 3b — §11).** All signing (`--sign`, Authenticode, codesign,
notarize, staple); macOS stub production; cross-building *from* a non-`x86_64-linux` host (the manifest
is baked only into the `x86_64-linux-gnu` primary in v1 — §6, P3-7).

## 2. Where it plugs in — the `build_stub` seam

`bundle/cross.rs::build_stub(target)` (today) is: resolve `cache_dir` (FNV-1a-64 of the running phorj
bytes) → **cache hit → return** → else **if no `Cargo.toml` → error** ("wait for Phase 3") → else
`cargo-zigbuild` from source → cache → return.

Phase 3a turns the miss path into a **3-way branch**:

```
build_stub(target):
  cached = cache_dir(phorj_bytes)/<triple>/phg[.exe]
  if cached.is_file():            return cached                      # unchanged
  if Path::new("Cargo.toml").is_file():
                                  return build_stub_local(target)    # Phase 2 cargo-zigbuild, unchanged
  else:                           return download_stub(target)       # NEW (distributed release)
```

Everything downstream of `build_stub` (`embed_section`, output naming, `build_target`/`build_all`/
`build_host`) is **unchanged** — a downloaded stub is interchangeable with a locally-built one, so the
produced artifact stays byte-identical to Phase 2's. The host build (`build_host`, reuses
`current_exe()`) is also unchanged: building for your own OS never downloads and stays offline.

## 3. The download-and-cache client (`download_stub`)

```
download_stub(target):
  1. manifest = manifest::active()                         # baked const, or PHORJ_STUB_MANIFEST override
  2. expected = manifest.lookup(target)                    # None -> precise error (no prebuilt stub)
  3. base     = manifest::registry_base()                  # PHORJ_STUB_REGISTRY, or repo+version default
  4. url      = base + "phg-stub-" + target (+ ".exe" if windows)
  5. create_dir_all(cached.parent());  tmp = cached.parent()/.download-<pid>   # same dir → same-fs rename
  6. fetch(url, tmp)                                        # §4 transport
  7. got = sha256::hex(read(tmp));  if got != expected:    # integrity gate (§5)
            remove(tmp); error "integrity check failed …"
  8. rename(tmp, cached)                                    # atomic publish into the cache
  9. return cached
```

**Verify-before-cache.** The hash is checked on the **temp** file; only a verified stub is moved into
the cache, so a corrupt/tampered/partial download never poisons the cache. A verified stub is cached
under the existing phorj-hash-keyed path, so a rebuilt phorj re-downloads (parity-spine guard, B-6).

**Failure modes (all precise, embed nothing, exit 1):**
- no manifest entry for `target` → `no prebuilt stub for '<target>' in phorj v<ver> — cross-building
  from this host needs a phorj source checkout`;
- `fetch` failed (curl error / missing file / non-200) → the transport error, prefixed with the URL;
- sha256 mismatch → `integrity check failed for '<target>' stub: expected <e>, got <g> — refusing to
  embed` (the parity-spine refusal);
- empty manifest (e.g. a dev-built binary, no `Cargo.toml`) → falls into the "no entry" case above.

## 4. Transport (`fetch`)

std has **no TLS** (`std::net` is raw TCP), so HTTPS download shells out to **`curl`** — host build
tooling, exempt from the std-only line exactly like `zig`/`cargo-zigbuild`/`llvm-objcopy` (master §6).
The artifact and runtime stay std-only.

- `http://` / `https://` base → `curl -fSL --proto =https,http -o <tmp> <url>` (`-f` fail-on-error,
  `-S` show errors, `-L` follow redirects — GitHub release assets 302 to a CDN). `PHORJ_CURL`
  override mirrors `PHORJ_OBJCOPY`. Missing curl → `curl not found — needed to download prebuilt
  stubs; install curl, or build from a phorj source checkout`.
- `file://` or a bare local path → `std::fs::copy` (no curl). This is the **hermetic test** path: a
  fixture-dir registry needs no network and no curl.

**Registry base.** Default `{CARGO_PKG_REPOSITORY}/releases/download/v{CARGO_PKG_VERSION}/` — derived
from a new `Cargo.toml` `repository` field and the compiled-in version. `PHORJ_STUB_REGISTRY` (runtime
env) overrides the base (trailing `/` normalised) — for hermetic tests, private mirrors, or pinning.
Asset name convention `phg-stub-<triple>[.exe]` is a shared constant used by both the client and CI.

## 5. Integrity — hand-rolled SHA-256 + the baked manifest

The artifact is an **executable**, and a wrong stub silently embeds your source into a mismatched VM —
breaking the parity spine (`docs/INVARIANTS.md` #1) at the distribution layer. So integrity is a real
security boundary here, unlike the FNV-1a-64 *cache key* (which is identity-only). It gets a real
cryptographic hash.

- **`bundle/sha256.rs`** — FIPS 180-4 SHA-256, hand-rolled std (~70 lines: the 64 round constants, the
  message schedule, the compression loop), `sha256_hex(&[u8]) -> String`. No `object`/`sha2`/`ring`
  crate — same hand-rolled-std ethos as the CRC-32 in `bundle/container.rs`. Known-vector tested
  (`"" → e3b0c442…b855`, `"abc" → ba7816bf…15ad`).
- **Manifest** — line-based text: `# …` comments and blank lines skipped; an optional `version <v>`
  line (sanity-checked against `CARGO_PKG_VERSION`); then `<triple> <sha256-hex>` lines. The parser is
  tolerant (whitespace-split, lowercase 64-hex validation; a malformed entry is skipped with a debug
  note, never a panic). Verified downloads must match exactly.

## 6. Manifest baking (`build.rs`) — and the circularity it breaks

The manifest holds the sha256 of each **cross stub**. But a stub *is* a phg binary; if the manifest
were `include_str!`'d into *every* binary, a stub's bytes would depend on the manifest, whose entries
are the hashes of those bytes — an unsolvable fixpoint.

**Resolution (P3-3).** A new **`build.rs`** reads the build-time env `PHORJ_BAKE_STUB_MANIFEST=<path>`:

- **set** (CI's primary build) → copy the file's contents to `$OUT_DIR/stub_manifest.txt`;
- **unset** (every other build) → write an **empty** `$OUT_DIR/stub_manifest.txt`.

The crate does `include_str!(concat!(env!("OUT_DIR"), "/stub_manifest.txt"))`. Consequences:

- **Cross stubs build with the env unset → empty manifest → their bytes are manifest-independent → their
  hashes are stable.** No fixpoint.
- **Dev builds** (env unset) get an empty manifest → the download path finds no entry → and a dev
  checkout has `Cargo.toml` anyway, so it builds locally. Correct by construction.
- **Only the CI primary build** (`x86_64-linux-gnu`) sets the env, so only the binary users install to
  cross-build carries the manifest (P3-7). Cross-building *from* Windows/aarch64 isn't supported in v1
  (those binaries have an empty manifest → the clear "needs a source checkout" error).

`build.rs` is host build tooling (runs at build time, never links into the artifact) — std-only line
intact. `cargo:rerun-if-env-changed=PHORJ_BAKE_STUB_MANIFEST` keeps rebuilds correct.

**Test seam.** A runtime env `PHORJ_STUB_MANIFEST=<path>` overrides the baked manifest, so integration
tests inject a fixture manifest without a release build. (Mirrors the `PHORJ_STUB_REGISTRY` seam and the
`_GS_EU2_HTTP_FIXTURE_DIR` philosophy elsewhere.)

*Alternatives considered & rejected:* a committed `include_str!` file that CI rewrites between passes
(works, but parks "CI rewrites me" release-data in source control and makes circularity-breaking a
manual CI discipline); post-build `.phorj-manifest` section injection (reuses the reader, but the
manifest then lives in a strippable section rather than `.rodata`). `build.rs`+env makes the
circularity break automatic and keeps generated data out of git.

## 7. CI stub registry (`.github/workflows/release.yml`)

GitHub Actions, triggered on a `v*` tag (and `workflow_dispatch`). **No secrets** — unsigned cross
stubs need no certs; publishing uses the Actions-provided `GITHUB_TOKEN`. Two passes break the
circularity:

1. **Build stubs** — a matrix over the 4 Phase-2 cross targets on `ubuntu-latest`: install `zig` (pinned
   0.16.0), `cargo-zigbuild`, the rustup target, and `llvm-tools-preview`; `cargo-zigbuild build
   --release --bin phorj --target <T>` with `RUSTFLAGS=--cap-lints=warn` (P2-6) and **`PHORJ_BAKE_STUB_MANIFEST`
   unset**. Each `target/<T>/release/phorj[.exe]` is a stub artifact.
2. **Manifest + primary** — gather the 4 stubs, compute each sha256 (CI uses host `sha256sum`; the
   *client* verifies with its hand-rolled SHA-256 — a cross-implementation check, §10/§13), write the
   manifest, then `cargo build --release` for `x86_64-linux-gnu` with `PHORJ_BAKE_STUB_MANIFEST` =
   that manifest → the released primary binary with the manifest baked.
3. **Publish** — create/update the release for the tag; upload assets `phg-stub-<triple>[.exe]`
   (the 4 stubs) + the primary `phg-x86_64-unknown-linux-gnu`. (Reproducible-build flags from master
   §5 are a nice-to-have; the sha256 manifest is the real integrity guard.)

The workflow is the one piece not fully exercisable here (it needs a real tag push), but it requires no
secrets, so it is runnable by the maintainer on the next release. It is lint-checked (`actionlint`/
`yamllint` if present).

## 9. CLI surface

**No user-visible flag change.** `phg build [--target T | --all]` is unchanged; the download is
transparent (a distributed phorj that errored in Phase 2 now succeeds). New **env vars**, documented:

- `PHORJ_STUB_REGISTRY` — override the registry base (default = repo releases for the running version).
- `PHORJ_CURL` — override the `curl` binary (mirrors `PHORJ_OBJCOPY`).
- `PHORJ_STUB_MANIFEST` — override the baked manifest (primarily a test seam).

## 10. Test strategy (mirrors Phase 2's tiers)

1. **Unit — offline, always run** (`#[cfg(test)]`):
   - `sha256_hex` known vectors (`""`, `"abc"`, a longer multi-block input).
   - `manifest::parse` — well-formed, comments/blanks, `version` line, malformed entry skipped, hex
     validation; `lookup` hit/miss; `registry_base` from `PHORJ_STUB_REGISTRY` vs default.
2. **Integration — offline, hermetic** (`tests/registry.rs`): a fixture-dir registry + a fixture manifest
   (`PHORJ_STUB_REGISTRY`/`PHORJ_STUB_MANIFEST`), driving `download_stub` from a temp dir with no
   `Cargo.toml`. A dummy blob covers the client logic without a real toolchain:
   - download copies the asset into the cache;
   - **sha256 mismatch → error, cache not poisoned** (tamper detection);
   - missing manifest entry → precise error; missing asset → precise error; empty registry → error.
3. **Real download→verify→embed→run — toolchain-gated, graceful skip** (`tests/registry.rs` or
   `tests/build.rs`): build a real `x86_64-unknown-linux-musl` stub via `cargo-zigbuild`, publish it
   into a fixture registry with its **real** sha256 (generated by host `sha256sum`), force the download
   path, then `embed_section` + execute the musl binary on this box → **byte-identical to `runvm`**
   (also a cross-implementation check: `sha256sum` ≡ hand-rolled SHA-256, or verification would fail).
   Skips with an `eprintln` if `cargo-zigbuild`/the musl target is absent (same pattern as Phase 2).
4. **CI workflow** — `actionlint`/`yamllint` if available; full exercise deferred to the first real
   release (no secrets required).

Gate unchanged: `cargo test` green, `cargo clippy --all-targets` + `cargo fmt --check` clean,
`#![forbid(unsafe_code)]`, zero new runtime crate deps.

## 11. Deferred / out of scope (Phase 3b)

- **All signing** — `--sign`; Windows Authenticode (`osslsigncode`); macOS codesign + notarize + staple
  (`rcodesign` from Linux). Needs certs the maintainer does not currently have; the trusted-cert paths
  would be unverifiable scaffolding (decision P3-1).
- **macOS stub production** — needs a macOS SDK for zig to link against; a Mach-O artifact can't be
  executed here. The Mach-O/fat *reader* already ships (Phase 2).
- **Cross-building from a non-`x86_64-linux` host** — the manifest is baked only into the `x86_64-linux`
  primary in v1 (P3-7); other hosts get the clear "needs a source checkout" error.

## 12. Decisions Log

| # | Decision | Choice | Rationale |
|---|---|---|---|
| P3-1 | Phase-3 split | **3a = registry only**; signing + macOS stub → 3b | No signing certs available → signing is unverifiable + unusable now; the registry is a complete, usable, fully-verifiable deliverable on its own (YAGNI on cert-gated code) |
| P3-2 | Download integrity | **Baked sha256 manifest** + hand-rolled std SHA-256 | Honors frozen B-6; the artifact is an executable, so a wrong stub breaks the parity spine at the distribution layer — a real crypto hash, tamper-evident even if the host is compromised |
| P3-3 | Manifest baking | **`build.rs` + `PHORJ_BAKE_STUB_MANIFEST`** (empty when unset) | Breaks the stub↔manifest circularity automatically (stubs build env-unset → manifest-independent bytes); keeps release-specific data out of git; runtime `PHORJ_STUB_MANIFEST` = test seam |
| P3-4 | Dev vs distributed | **`Cargo.toml` presence** (reuses P2-9) | A source checkout builds locally; a sourceless release binary downloads |
| P3-5 | Transport | **`curl`** for http(s) (host-tool, std-exempt) + `fs::copy` for file/local | std has no TLS; consistent with the zig/objcopy exemption; the local path makes tests hermetic |
| P3-6 | Registry base | `CARGO_PKG_REPOSITORY` + version; `PHORJ_STUB_REGISTRY` override | Single-sources the repo URL (new `repository` field); the override enables hermetic tests + private mirrors |
| P3-7 | Manifest reach (v1) | Bake only into the **`x86_64-linux` primary** | Breaks mutual circularity; the primary dev host is the only cross-build origin needed now |
| P3-8 | CI | GitHub Actions, **2-pass, no secrets**; `GITHUB_TOKEN` to publish | Unsigned cross stubs need no certs; the maintainer has Actions; 2-pass is the circularity break in CI form |

## 13. Risks

- **Hand-rolled SHA-256 correctness** — mitigated by FIPS-180-4 known vectors *and* the tier-3 real
  download, where the CI-style `sha256sum`-generated manifest must agree with the in-binary hand-rolled
  hash or verification fails (a built-in cross-implementation check).
- **`curl` absence on a user machine** — precise actionable error; the host build (no download) still
  works offline.
- **GitHub asset URL / naming drift** — the `phg-stub-<triple>` name + the `releases/download/v<ver>/`
  path are shared constants between client and CI; a rename touches both in one commit.
- **Manifest/version skew** — the `version` line is sanity-checked, and the FNV cache key already
  invalidates stubs across phorj versions; a downloaded stub from the wrong version fails the sha256
  check.
- **CI not fully exercisable here** — the workflow needs a real tag push; it needs no secrets, so the
  maintainer runs it on the next release. Until then it is lint-only.
- **Reproducibility** — the downloaded stub must equal the CI-built bytes; the sha256 manifest enforces
  exact equality (stronger than relying on reproducible-build flags alone).
