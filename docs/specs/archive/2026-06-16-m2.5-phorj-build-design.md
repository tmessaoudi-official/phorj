# Phorj M2.5 — `phg build` (standalone native executables) — Design

> Milestone M2.5 of the roadmap: turn a `.phg` program into a **standalone native executable** for
> every supported OS. M2 (`docs/MILESTONES.md`) is complete; this is the immediate next slice.
> Frozen design; the Phase-1 implementation plan lives in `docs/plans/`. Reviewed by an independent
> advisor (two passes) — the end-state below incorporates that review.

## 1. Goal & Non-Goals

**Goal.** `phg build foo.phg` produces a self-contained executable that runs `foo.phg` on the VM
with no Phorj install on the user's machine — for Linux (gnu + musl, x86_64 + aarch64), Windows
(x86_64), and macOS (x86_64 + aarch64). The produced binary stays **std-only / zero-dependency**.

**Non-goals.**
- New language features (argv access, custom exit codes, file I/O) — these are **M3**. v1 artifacts
  ignore argv and always exit 0 on success (documented limitation).
- A bundler/linker of multiple `.phg` files — there is no module system yet (M5). One program in,
  one binary out.
- Replacing the transpiler's PHP path — orthogonal.

## 2. The unifying decision — payload is a named **section**, never an appended overlay

A raw overlay (bytes appended after EOF + a footer) is the cheap approach and works on ELF/PE today,
but it is a **dead end on Mach-O**: arm64 macOS mandates a code signature at exec, the signature must
be the last content in the file, and anything appended after it invalidates the signature. Therefore:

**The payload is always stored as a named section in the host object format** — `.phorj` on ELF and
PE/COFF, `__PHORJ,__source` on Mach-O — added with `llvm-objcopy --add-section` (from rustup's
`llvm-tools-preview`, which handles all three formats from a Linux host). A section lives *inside* the
signed region, so the same mechanism is Authenticode- and `codesign`-compatible on every format. The
invariant is **section-first, sign-last**.

This gives one uniform model: *locate a named blob inside my own image → validate → run it*; only the
**retrieval** (object-format parsing) differs per OS, and that parser is hand-rolled std (no `object`/
`goblin` crate — see §6).

## 3. The payload container (forward-compatible)

The section holds a **versioned, CRC-guarded container**, not raw source — so today's "embed source"
can become "embed bytecode", gain argv/exit-code metadata, or add compression without a format break.
Little-endian throughout (all six targets are LE):

```
PHORJ PAYLOAD CONTAINER  (inside .phorj / __PHORJ,__source)
 off  size  field
   0     8  magic             = "PHORJ\0\0"
   8     2  container_version = 1            (bump on incompatible layout)
  10     2  header_len        = 32           (header may grow; old readers skip unknown tail)
  12     1  payload_kind      0=source_utf8  1=bytecode(future)  2=reserved
  13     1  compression       0=none         1=deflate(future)   2=zstd(future)
  14     1  encryption        0=none         1=future
  15     1  flags             bit0=has_argv_spec  bit1=has_exitcode  (reserved)
  16     8  payload_len       (u64)
  24     4  payload_crc32     (CRC-32 of the stored payload bytes)
  28     4  header_crc32      (CRC-32 of bytes [0..28))
  32   var  payload           (source UTF-8 in v1)
```

Reader contract: locate section → absent ⇒ behave as the normal `phorj` CLI (fall-through) → check
`magic` → check `header_crc32` (guards against trusting a garbage `payload_len`) → bounds-check
`payload_len` against the section size → check `payload_crc32` → refuse `container_version` newer than
known ("artifact built for a newer phorj") → dispatch on `payload_kind`. A `current_exe()` failure or
any validation failure **falls through to the normal CLI; never panics**. CRC-32 is hand-rolled std
(table-driven, ~10 lines).

**v1 payload = the `.phg` source text** (`payload_kind=0`). The binary already contains
lex→parse→check→compile→VM, so it re-runs the source on the VM at startup (~17µs parse+check+compile
per `phg bench` — negligible). This avoids a fragile hand-rolled bytecode serializer that would
have to track every `Op`/`Value`/`EnumDesc`/`ClassDesc` change (a fourth coupled match surface beyond
the three in `docs/INVARIANTS.md`). The source→bytecode migration is later a `payload_kind` flip, not
a format change. **Documented limitation:** the source is recoverable from the artifact (`strings`/a
section dump) — acceptable for v1; bytecode (`payload_kind=1`) raises the bar later.

## 4. Per-target embedding mechanism (end state)

| Target | Format | Section | Added via | Signing path |
|---|---|---|---|---|
| `x86_64-unknown-linux-gnu` | ELF | `.phorj` | `llvm-objcopy --add-section .phorj=… --set-section-flags .phorj=noload,readonly` | none required; optional SHA256SUMS + GPG on release |
| `aarch64-unknown-linux-gnu` / `-musl` | ELF | `.phorj` | same | same |
| `x86_64-unknown-linux-musl` | ELF | `.phorj` | same | same (static; cleanest zig target) |
| `x86_64-pc-windows-gnu` | PE/COFF | `.phorj` | same `llvm-objcopy` | Authenticode (EV cert) on the *final* artifact via `osslsigncode`; a real PE section is covered by the Authenticode hash and avoids overlay AV heuristics |
| `x86_64-apple-darwin` / `aarch64-apple-darwin` | Mach-O | `__PHORJ,__source` | `llvm-objcopy --add-section`, **then** sign | `rcodesign` (ad-hoc `-s -` to run locally; Developer-ID + notarize + staple to distribute) |

Caveat baked into the design: because the embedded payload changes the file hash, **the *final*
artifact on PE/Mach-O cannot be pre-signed by CI** — CI signs the *stub* (meaningful only for Linux);
final-artifact signing is the opt-in `--sign` step (§5).

## 5. Build orchestration — the stub-registry model

`phg build` does **not** invoke a compiler at build time in the end state. Instead:

1. **CI (GitHub Actions), once per phorj release:** builds one **runtime stub** per target, signs/
   notarizes where applicable, publishes them as release assets (`phg-stub-<target>-<version>`)
   with a hash manifest **baked into the phg binary**.
2. **`phg build foo.phg [--target T | --all] [--sign …]`:**
   - **Validate first** — lex→parse→check; on error, fail with diagnostics (exit 1), emit nothing.
   - Resolve target(s): host default; `--target <triple>`; `--all`.
   - Fetch the stub from the local cache `~/.cache/phorj/stubs/<phorj-version>/<target>`; if absent
     and online, download the release asset and verify its hash against the baked-in manifest. The
     **host-target stub ships inside the phorj release tarball**, so building for your own OS is
     offline from first run.
   - Copy stub → output; `llvm-objcopy --add-section` the container; `chmod +x` (unix).
   - **Optional `--sign`** shells out to the platform signer (`rcodesign` / `osslsigncode`) with
     user-supplied credentials to produce a distributable signed artifact. Default = runnable,
     unsigned (ad-hoc on macOS so it launches).
   - Output: host → `./foo` (`foo.exe` for windows); `--all` → `dist/foo-<target>[.exe]`.

**Cache key includes the phg binary's own hash**, not just the triple — otherwise a stale cached
stub would embed source that runs on an *old* VM, silently breaking the parity spine
(`docs/INVARIANTS.md` #1) at the distribution layer.

**Signing without a Mac:** `rcodesign` performs macOS sign + notarize (App Store Connect API key over
HTTPS) + staple **from a Linux runner** — no Mac in the pipeline (a real Mac is a nice-to-have for
final smoke tests only). Windows Authenticode uses an EV cert in CI secrets.

**Reproducibility:** stub builds set `SOURCE_DATE_EPOCH` + `--remap-path-prefix` + sorted sections;
`phg build`'s only mutation (`--add-section` of a fixed payload) is deterministic, so the final
artifact is reproducible given a fixed source + stub.

## 6. The std-only ethos boundary

**Rule:** *the produced binary and the phg runtime crate stay std-only / zero runtime deps;
everything that builds, embeds, signs, or ships an artifact is host build-tooling and is exempt.*

- **Inside the line (std-only):** the phg binary, VM, the **hand-rolled object-format reader**
  (ELF/PE/Mach-O/fat section lookup), the container + CRC-32 logic. No `object`/`goblin`/`serde`.
- **Outside the line (exempt host tooling):** `zig` / `cargo-zigbuild`, `llvm-tools` (`llvm-objcopy`),
  `rcodesign`, `osslsigncode`, GitHub Actions, `notarytool`. Invoked as subprocesses / used at build
  time; none links into the artifact.
- **Test:** *does it end up linked into the 1.1 MB artifact or its runtime?* No → exempt.
  **Leakage risk to watch:** the temptation to pull in `object`/`goblin` to *read* the section (that
  code runs inside the artifact) — forbidden; the reader is hand-written std.

## 7. Phased roadmap (each phase a strict subset of the same architecture — no rework)

**Phase 1 — runnable & verifiable on this Linux box today (no CI, no signing):**
- Implement the **complete container format** (§3): magic, version, `header_len`, both CRC-32s,
  bounds-check.
- Implement the **ELF section reader** with the dispatch-on-magic skeleton present (only the ELF arm
  wired) and graceful `current_exe()` fall-through to the normal CLI.
- `phg build` for **`x86_64-unknown-linux-gnu` only**, via `llvm-objcopy --add-section .phorj=`,
  payload = source text, cache key = phorj-binary hash. Host stub = the locally-built phorj.
- Verifiable now: build → run → `llvm-objcopy --dump-section` round-trip, CRC-tamper rejection, and
  fall-through (a phorj with no `.phorj` section behaves as the normal CLI).

**Phase 2 — multi-target cross-build from Linux (still no signing infra):**
- Fill in the PE / Mach-O / fat reader arms (skeleton from Phase 1).
- Cross-compiled stubs via zig / `cargo-zigbuild` for aarch64-linux (gnu+musl), x86_64-linux-musl,
  x86_64-pc-windows-gnu; glibc floor pinned for gnu targets. Same `--add-section` embedding.
- macOS: ad-hoc-signed artifacts via `rcodesign sign -s -` (exercises section-first/sign-last).
- Local **stub cache** keyed on stub+phorj hash.
- Build stubs with `--cap-lints=warn` for the stub-build invocation so target-specific lints don't
  trip the `[lints] warnings = "deny"` manifest gate (which cannot be relaxed per-target without
  editing the tracked `Cargo.toml`).

**Phase 3 — registry + full signing/notarization (CI lands):**
- GitHub Actions matrix builds all six stubs per release, publishes assets + a hash manifest baked
  into phorj.
- Windows Authenticode (EV cert) and macOS Developer-ID + notarize + staple via `rcodesign` on a
  Linux runner.
- `phg build` flips to download-and-cache; `--sign` opt-in for end-user final-artifact signing.

## 8. Success criteria

- **Phase 1 (gates M2.5 v1):** `phg build examples/realworld/ledger.phg` on this host produces an
  ELF that, when executed, prints byte-identically to `phg runvm examples/realworld/ledger.phg`;
  the container round-trips; a tampered CRC or truncated section falls through to the CLI without
  panicking; a phorj with no section behaves exactly as today. `cargo test` green, clippy + fmt
  clean, `#![forbid(unsafe_code)]` intact, zero new runtime crate deps.
- **End state:** all six targets produced; macOS/Windows artifacts signable from Linux; offline build
  for cached/host targets; the parity spine holds at the distribution layer (cache keyed on phorj
  hash).

## 9. Risks

- **Hand-rolled object readers** (PE/Mach-O/fat) are non-trivial — mitigated by phasing (ELF first,
  fully tested) and by needing only *section lookup*, not full parsing.
- **zig 0.16.0 is pre-1.0** and its `zig cc` triple/flag handling has churned — Phase 2 pins the zig
  version and prefers `cargo-zigbuild` (host build-dep) over a bespoke wrapper for windows-gnu/glibc.
- **`rustup target add` / stub download need network** — mitigated by shipping the host stub in the
  release tarball and a precise offline error otherwise.
- **`warnings = "deny"` vs cross builds** — mitigated by `--cap-lints=warn` on the stub-build
  invocation (verified against 1.96 in Phase 2).

## 10. Decisions Log

| # | Decision | Choice | Rationale |
|---|---|---|---|
| B-1 | Embedding mechanism | **Named section** (`.phorj` / `__PHORJ,__source`), not overlay | Only section-embedding is signing-compatible on Mach-O (arm64 mandatory sign); uniform across formats |
| B-2 | Payload contents (v1) | **Source text** (`payload_kind=0`) | Binary already has the full pipeline; avoids a fragile bytecode serializer (a 4th coupled match surface) |
| B-3 | Payload container | Versioned + dual-CRC, `payload_kind`/`flags`/`compression` reserved | Forward-compat: source→bytecode and argv/exit-code are value flips, not format breaks |
| B-4 | Section tooling | `llvm-objcopy` (rustup `llvm-tools-preview`) | One tool, all three formats, from Linux |
| B-5 | Orchestration | **Stub-registry** (CI builds/signs stubs once per release; `phg build` fetches+caches+embeds) | Instant builds, offline-once-cached, signing-ready by construction (bun/deno model) |
| B-6 | Cache key | Includes the phorj **binary hash**, not just the triple | A stale stub would embed source running on an old VM — breaks the parity spine at distribution |
| B-7 | macOS signing | `rcodesign` from a Linux runner (sign + notarize + staple) | No Mac in the pipeline; Apple Notary API is HTTPS |
| B-8 | Final-artifact signing | Opt-in `--sign`; default runnable/unsigned (ad-hoc on mac) | Payload changes the hash, so CI can't pre-sign the final PE/Mach-O artifact |
| B-9 | std-only boundary | Artifact + runtime zero-dep (hand-rolled reader); build tooling exempt | Test = "does it link into the artifact?"; forbids `object`/`goblin` in the reader |
| B-10 | Phasing | ELF-only Phase 1 → cross-targets Phase 2 → CI+signing Phase 3 | Each phase a strict subset of the same section+container architecture; no rework |
| B-11 | argv / exit code | Not supported in v1 (documented) | The language has no argv/exit surface yet (M3) |
