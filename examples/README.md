# Phorj examples

What Phorj can do **today**. Every `.phg` here runs byte-identically on both backends
(the bytecode VM and the tree-walking interpreter oracle) ‚Äî enforced by `tests/differential.rs`, which globs this directory,
so a new example is auto-gated the moment it lands. This page is updated as examples are added.

## Sharp edges

- **Every file declares a package (M5 S1) ‚Äî `package Main;` is the runnable entry.** Nothing lives
  "in the wind": each file's first line is a `package` declaration, never inferred. A runnable program
  uses the reserved `package Main;` (every example here starts with it); `Core` is reserved for the
  stdlib. Dotted library packages (`package Acme.Convert;`) + strict folder=path + cross-package
  imports are now **shipped** ‚Äî see `project/tempconv/` and `project/README.md`. Package and
  folder segments are **PascalCase** (`E-PKG-CASE`), mapping 1:1 to PHP namespaces (`Acme.Convert`
  ‚áí `Acme\Convert`); types are PascalCase, functions/variables camelCase.
- **Zero-payload enum variants use call form `V()` everywhere** ‚Äî to construct (`Defend()`) *and* in
  a `match` arm (`Defend() =>`). A bare `Defend =>` arm is a catch-all *binding*, not a variant
  pattern, so it silently swallows every case.
- **`import Core.Output;` is load-bearing (M3 Wave 1).** Everything is namespaced ‚Äî "nothing in the
  wind" ‚Äî so there is no free global `printLine`: a program must `import Core.Output;` and call
  `Output.printLine(...)`. Stdlib modules are reserved under `core.*`; the root lives in the import and
  the leaf qualifies the call (Go's `import "fmt"` ‚Üí `fmt.Println`). The same leaf-qualified `import`
  resolves user `.phg` packages in a project (M5) ‚Äî see `project/tempconv/`.
- **Member visibility is enforced ‚Äî the rejected accesses `guide/member-visibility.phg` can't show.**
  Each of these is a compile error (so it cannot appear in a runnable example), demonstrated by the
  guide's *legal* counterparts instead:
  - reading a `private` field from outside its class ‚Äî `var c = new C(5); var x = c.secret;`
    ‚Üí `E-FIELD-VISIBILITY` (use a public accessor: `c.balanceOf()`);
  - writing a `private`/`protected` field from outside ‚Äî `c.secret = 9;` ‚Üí `E-FIELD-VISIBILITY`;
  - calling a `private`/`protected` method from outside ‚Äî `c.audit();` ‚Üí `E-METHOD-VISIBILITY`;
  - reading a `protected` member from a non-subclass (e.g. `main`) ‚Äî ‚Üí `E-FIELD-VISIBILITY`.
  `phg explain E-FIELD-VISIBILITY` / `E-METHOD-VISIBILITY` document each.

- **Constructor visibility is enforced ‚Äî the rejected `new` `guide/ctor-visibility.phg` can't show.**
  - external `new Counter(0)` on a `private constructor` (from `main`) ‚Üí `E-CTOR-VISIBILITY`
    (construct via a static factory or singleton field ‚Äî both run in the class's scope);
  - a `protected constructor` constructed from outside its class/subclasses ‚Üí `E-CTOR-VISIBILITY`;
  - a non-visibility modifier on a constructor ‚Äî `static constructor(...)` ‚Üí `E-CTOR-MODIFIER`.
  `phg explain E-CTOR-VISIBILITY` / `E-CTOR-MODIFIER` document each.

- **Definite assignment is enforced ‚Äî the rejected `guide/field-safety.phg` can't show.**
  - a non-optional field never assigned ‚Äî `class C { int x; constructor() {} }` ‚Üí `E-FIELD-UNINITIALIZED`;
  - a field assigned only in one branch ‚Äî `if (flag) { this.x = 7; }` ‚Üí `E-FIELD-UNINITIALIZED`
    (not 'every path'). Assign unconditionally, give an initializer, promote it, or make it optional.
  `phg explain E-FIELD-UNINITIALIZED` documents it.

- **Static methods can't touch instance state ‚Äî the rejected `guide/static-methods.phg` can't show.**
  - `static function f() -> int { return this.x; }` ‚Üí `E-STATIC-THIS` (no instance in a static method);
  - a bare instance-field read in a static method ‚Äî `return x;` ‚Üí `E-STATIC-THIS`.
  Access static members as `Class.member`, pass values as parameters, or drop `static`.
  `phg explain E-STATIC-THIS` documents it.

- **Overloads of one name must agree on `static`-ness (Statics-B) ‚Äî no runnable example (a compile
  error).** A method name that has both a `static` and an instance overload ‚Äî
  `class C { static function f(int x) -> int {‚Ä¶} function f(string s) -> int {‚Ä¶} }` ‚Üí
  `E-OVERLOAD-STATIC-MIX`. A mixed set has no sound call form (`C.f(..)` would resolve only the static
  overloads, `x.f(..)` only the instance ones), and PHP forbids it too. Make every overload `static`,
  or none. `phg explain E-OVERLOAD-STATIC-MIX` documents it.

- **Declaration names must be unique (Soundness Batch G)** ‚Äî no runnable example (these are all
  compile errors):
  - duplicate parameter names ‚Äî `function add(int a, int a)` ‚Üí `E-DUP-PARAM` (free fn, method, ctor);
  - duplicate explicit field names ‚Äî `class C { int x = 0; int x = 1; }` ‚Üí `E-DUP-FIELD`.
  Previously the later declaration silently won. An explicit field that also names a promoted ctor
  param stays allowed (the explicit declaration is authoritative). `phg explain E-DUP-PARAM` /
  `E-DUP-FIELD` document each.
